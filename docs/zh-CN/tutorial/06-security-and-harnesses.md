## 三条安全底线（写进团队 Wiki 的那种）

1. **密钥与令牌**：永不写进仓库、教程或截图；用环境变量或秘密管理器。教程与 PR 审查中若发现示例像真密钥，一律视作事故并完成轮换。  
2. **敏感变更仍需人工 gates**：认证、支付、权限模型、导出个人数据等路径，ECC 可以帮你写代码与检查清单，**不能替代**你的安全评审与发布审批。  
3. **「装了 ECC」≠「已通过安全审计」**：它提供规则、技能与可选扫描命令；合规与威胁模型仍由你的组织负责。

## MCP 与外向数据

**MCP** 配置在 [`mcp-configs/`](../../../mcp-configs/)，描述「助手可以接哪些外部队列 / 服务器」。教学时注意：

- 正文与截图只使用 **占位符**（如 `YOUR_API_KEY`）；  
- 发现误提交：立即轮换密钥 + 从历史中清除（按组织流程）。

MCP 扩大能力的同时也扩大 **误用外联数据** 的风险；与 **最小权限原则** 一起设计（只开需要的工具、只给需要的 scope）。

## 多 harness 再强调（与安全相关）

| 注意点 | 说明 |
|--------|------|
| 规则注入方式不同 | 各 IDE 如何把 `rules/`、`AGENTS.md` 注入上下文可能不同，**敏感表述**（如「禁止执行某类 Bash」）要在每端验证是否真的生效。 |
| Hook 不等价 | 见第 4 篇：依赖事件的自动化在迁移 harness 时**必须重测**。 |
| MCP 配置分散 | Cursor 等在 UI 里配的 MCP 与仓库内 `mcp-configs/` **可能两套并存**；避免重复定义相互矛盾的权限。 |

更新本教程时：改 **元信息日期**，并抽检 `plugins/` 下说明是否仍指向正确路径。
