## Hook 是什么

**Hook（钩子）** 是在特定 **生命周期事件** 或 **工具调用前后** 自动执行的脚本或提示逻辑。和你主动输入 `/xxx` **不同**：Hook 由 harness 在「即将执行 Bash」「刚跑完编辑」「本轮回复结束」等时机触发。

设计目的包括：提醒（例如在长跑命令前提示开 tmux）、轻量校验、把上下文写回磁盘、在压缩前做摘要等。具体能力取决于 **harness 是否实现对应事件**——换用 Cursor 等时，部分钩子可能**无声失效**或表现不同，见排错文档。

钩子类型的名目与直觉解释与 [简明指南中的「钩子」一节](../the-shortform-guide.md) 对齐；下面用**各一句话**帮助记忆：

| 时机（常见英文名） | 一句话场景 |
|--------------------|------------|
| PreToolUse | 某个工具（如 Bash）**将执行前** —— 可做风险提示或拦截思路。 |
| PostToolUse | 工具**执行完毕后** —— 可格式化输出、写日志、轻量反馈。 |
| UserPromptSubmit | 你**发出一条用户消息时** —— 可注入短提醒或分类。 |
| Stop | **本轮助手回复结束**时 —— 适合做会话摘要或收尾写入。 |
| PreCompact | 上下文**压缩前** —— 保留关键状态、触发「战略压缩」类 skill。 |
| Notification | 与权限、通知相关的交互点（依 harness 而定）。 |

## 实验 C：阅读一条 Hook

1. 在仓库中打开 [`hooks/`](../../../hooks/)，任选 **一个** `.json` 配置（或主配置文件中被 include 的片段）。  
2. 找到 **matcher**（或等价条件）：它决定「在什么条件下这条 hook 会考虑运行」。  
3. 找到 **hooks** 数组里的 **command**、**notification** 等动作描述。  
4. 用**一句话**写出：*这条 hook 大概在什么时候想达成什么目的？*

可选：阅读 [`hooks/README.md`](../../../hooks/README.md) 了解本仓库钩子的组织方式。

**验收**：你能向同事**口头说明**这条 hook 的触发意图，而不是只念 JSON 字段名。

## Rules 与长驻约束

**Rule** 通常放在 [`rules/`](../../../rules/)，并与根目录 [`CLAUDE.md`](../../../CLAUDE.md) 一起构成「主会话默认应遵守」的边界：编码风格、安全底线、测试期望、编排习惯等。它们与 **Skill** 的差别可以粗暴记为：

- **Rule / CLAUDE.md**：偏 **Always-on** —— 像「家规」，只要在这个仓库里工作就应顾虑。  
- **Skill**：偏 **按需加载** —— 像「操作手册」，做某类任务时才完整展开。

---

## 实验 D：Rule 与 Skill 对比

1. 打开 [`rules/`](../../../rules/) 下**任意一则**与你有关的规则（或 [`CLAUDE.md`](../../../CLAUDE.md) 中的一段强约束）。写 **2～4 句**：它要求会话**始终**记住什么？  
2. 打开 [`skills/`](../../../skills/) 下**任意一则** skill（或子目录中的 SKILL.md）：写 **2～4 句**：它建议在**什么场景**下打开、分几步做？  
3. **验收**：你能用一句话总结 **「长驻 vs 按需」** 在你举的两例中如何体现。

篇幅不必长；对比协议论文不是本篇目标。

## 常见坑

- **Hook 写了从不触发**：先问 harness 是否支持该事件，再查 matcher 是否过窄。  
- **把复杂业务规则全塞进 Hook**：Hook 宜保持短、稳、可观测；重逻辑放代码与测试。  
- **在 A  harness 抄 B 的 hook 路径**：以目标工具的文档与本仓库 `plugins/` 为准。
